# Decisions

> Forward-looking design-decision register. Append-only. Newest entries on top.
>
> **What belongs here:** choices made with reasoning that will outlive the moment they were made. Not "we shipped X" — that goes in `_Change_Log.md`. This is the *why*: why X over Y, what was considered, what was rejected.
>
> **What does NOT belong here:** stray ideas (→ `08_Ideas_Parking_Lot/`), open questions (→ `09_Open_Questions/`), shipped change records (→ `_Change_Log.md`), session recaps (→ `07_Sessions/`).
>
> **Lifecycle:** entries are written by Claude during end-of-session integration when a session involved a decision. They never get edited or deleted — if a decision is later reversed, append a new decision that supersedes the old one.
>
> **Why this file exists separately from the change log:** the change log records *what changed*; this file records *what choice was made*. Sometimes both apply (a decision shipped); sometimes only the decision exists (we picked an approach for a future build). Future-Claude needs both records, but for different reasons.

2026-04-29 — `fell_asleep_here` uses idle detection (`Player.Idled`), not focus detection (`AfkDetector`)
- Decision: The `fell_asleep_here` achievement triggers off Roblox's server-side `Player.Idled` event (default ~20 minutes of mouse/keyboard inactivity), not the existing focus-based `AfkDetector` that fires on `WindowFocused` / `WindowFocusReleased`.
- Why: The achievement title's intent is "the player physically walked away from the keyboard" — fell asleep, called away, abandoned the session. Focus-loss doesn't capture that — it just means "switched windows," which any normal player does dozens of times per session (Discord, chat, looking up info). Firing the title on focus-loss would reward "alt-tabbed once," cheapening the title's loyalty/wear-yourself-out connotation. `Player.Idled` is the closer semantic match: it fires when the player has been mouse/keyboard-inactive for 20 minutes, which crosses the spec's 19-minute target with margin. Server-readable directly (no client-side timer to spoof or maintain). The existing `AfkDetector` stays in place for its actual purpose (XP rate decisions) — the two systems just operate on different definitions of "away."
- Alternatives considered: (a) reuse `AfkDetector` because it's already wired — rejected on semantics; the AFK rate state is also intentionally generous about focus-loss (per the 2026-04-27 sitting-overrides-AFK decision) which would skew the title's threshold further from intent. (b) implement a custom server-side input-timestamp tracker — rejected as more code than `Player.Idled` warrants; the title is cosmetic, not correctness-critical, so the standard 20-minute Idled threshold is fine. (c) drop the title from this brief and wait for a stronger idle/inactivity story — rejected; the title fits the Achievement category's loyalty-seed slot well, and `Player.Idled` is good enough for a v1.
- Lives in: `02_Systems/Title_System.md` § "Open Questions (resolved)" entry; `06_Codex_Plans/2026-04-29_AchievementTracker_v1.md` § 2 trigger table + § 5 step 14; the implemented `player.Idled:Connect` line in `src/ServerScriptService/Progression/AchievementTracker.lua`.
- Reversibility: trivial — the connection is one line. Swapping to focus-based detection or a custom timer would be a small follow-up brief if the 20-minute threshold turns out to misfire (e.g. on console where Idled may behave differently).

2026-04-29 — `heard_them_all` ships now with min-NPC gate (3); future-proofs against NPC roster growth
- Decision: The `heard_them_all` achievement is implemented in this brief but is gated by a `TitleConfig.HEARD_THEM_ALL_MIN_NPCS = 3` constant. The title only fires when (a) the player has talked to every NPC currently registered in `DialogueData.Characters` AND (b) the registry contains at least 3 entries. With QuietKeeper as the only NPC, condition (b) holds the title dormant. When Tyler adds NPCs 2 and 3, the gate opens automatically — no code change needed.
- Why: Tyler asked to ship the achievement now AND future-proof against new NPCs ("we are not finished with NPC systems"). Two interpretations of "future-proof": (i) ship it with auto-completing-after-one-conversation behavior and accept that early players get the title cheaply, or (ii) ship the mechanic but gate it so it stays dormant until the roster justifies the title's "every voice" framing. Option (ii) preserves the title's value: when a future player earns `heard_them_all`, it actually means they talked to a meaningful number of distinct characters, not "they happened to bump into the only NPC in the world." The persistent `npcsHeard` set means players who talk to QuietKeeper today get partial credit forever — the gate just delays firing until the world has more voices to hear. The threshold value (3) is tunable in `TitleConfig` if Tyler later wants to bump it to 5 or 7 — single-config-line change, no logic edits.
- Alternatives considered: (a) hold the title back from this brief entirely, ship it when NPC #2 arrives — rejected; would mean another full brief for one title, more overhead than gating. (b) ship without the gate, accept auto-completion — rejected per above; cheapens the title. (c) hardcode the threshold inside AchievementTracker rather than TitleConfig — rejected; threshold is a tuning lever, belongs in config alongside related constants like `LAUNCH_WINDOW`.
- Lives in: `06_Codex_Plans/2026-04-29_AchievementTracker_v1.md` § 11 "min-NPC gate is the future-proof lever" notes; `src/ReplicatedStorage/TitleConfig.lua` `HEARD_THEM_ALL_MIN_NPCS` constant; `src/ServerScriptService/Progression/AchievementTracker.lua` `checkHeardThemAll` function.
- Reversibility: trivial — change the constant, or remove the gate check, or change `min < 3` to `min < N`. Single-line edits.

2026-04-29 — `knows_every_chair` is per-server-session, not per-life
- Decision: The `knows_every_chair` achievement (sit at 5 different SeatMarker seats) tracks the player's unique seat set in server memory keyed per-session. The set persists across respawns within the same play session but resets when the player rejoins. Once earned, the achievement flag is persisted in TitleData like any other.
- Why: Tyler picked "tracked in memory per session" during the design pass. Per-session means harder than persistent-across-rejoins (the player must visit 5 distinct seats in one play session, not accumulate them over weeks of casual sitting), which fits the title's "knows every chair" connotation — feels like a deliberate explore-the-cave act, not a passive accumulation. Per-life (resetting on respawn) would be punitive and contrary to Tyler's earlier clarification that "per session" means server-session. Implementation: the per-player set lives in AchievementTracker's `_sessionState[player]` table and clears on `PlayerRemoving` only; respawn handlers don't touch it.
- Alternatives considered: (a) persistent across rejoins — rejected per above; cheapens the achievement. (b) per-life (reset on respawn) — rejected; punishes normal play patterns where players die or `/re` mid-session. (c) per-session as decided — chosen.
- Lives in: `06_Codex_Plans/2026-04-29_AchievementTracker_v1.md` § 5 step 12 implementation note; `src/ServerScriptService/Progression/AchievementTracker.lua` `_sessionState` + `disconnectPlayer` cleanup paths.
- Reversibility: trivial — move `state.seatKeys` from `_sessionState` (per-session) into `titleData.seatKeysSeen` (persistent). One small edit. If Tyler observes that 5 unique seats per session is too steep against the testing-place geometry (currently 8 seats exist), bump `SEAT_COUNT_TARGET` down or relax to persistent.

Polished TitleMenu (which now also covers the nametag title visual) and v1 retirement run as parallel Tyler-led tracks; polished menu best slots after Achievement so it has 12 more titles to display when it gets its polish pass. (b) Presence first because it's the cheapest — rejected; cheapness is a cost factor, not a value factor, and Presence on its own is just a counter — pairing Achievement first means the player is in the rhythm of "things being noticed" before time-spent titles tick. (c) Exploration first because it ties to the existing Discovery XP source — rejected; Exploration is gated on world content that doesn't exist yet. (d) Seasonal first to get the seasonal content engine built early — rejected; building seasonal infrastructure for a return arc that's still leaky is investing in the loyalty endgame before the return arc has loyal players to receive it.
- Lives in: `02_Systems/Title_System.md` § Category Activation Sequence (the canonical sequence with rationale); `00_Index.md` Active Systems annotation on Title_System pointing at the section; the future Codex briefs that will activate each category in turn.

2026-04-28 — QuietKeeper_Memory MVP gates on visit count only; full data captured from day 1 to avoid migration
- Decision: The MVP build of `QuietKeeper_Memory` only *uses* the `visitCount` field for dialogue gating, but the system *captures* all 8 fields (visit count, last visit, current streak, longest streak, total seated seconds, first visit, recent unlocks, has-been-outside) from day one. The remaining fields are dormant in the MVP and become surfaces in v2+ as the corresponding dialogue variants get authored.
- Why: Two competing pressures. (1) MVP scope discipline says ship the smallest meaningful slice — visit count alone is the cheapest meaningful gate. (2) NPC memory data has a long tail; once the system is shipped it's painful to add new fields later because every existing player's profile would need a migration. Capturing all 8 fields from MVP day one means v2 expansion (gap detection, streak detection, title acknowledgement, seasonal awareness, hasBeenOutside-aware lines) requires no migration — the data is already there, only the surface needs authoring. The cost of capturing dormant fields is small (a few extra fields in the DataStore payload, no runtime cost since nothing reads them in MVP); the cost of NOT capturing them and having to migrate later is large (one-shot migration code, runtime tests, rollback plan, soak period — all of which we just experienced in the Title v2 → v1 migration arc, PRs #14 + #17). Same lesson, applied earlier this time: data shape stability ahead of surface readiness.
- Alternatives considered: (a) only persist the fields the MVP uses; add new fields when v2 surfaces ship — rejected; would require a player-data migration for each new variant added later, which is a poor trade against capturing a few unused fields up front. (b) define every variant up front (gap, streak, recent unlocks all wired in MVP) — rejected; over-scopes the MVP brief, makes the first build harder to review and ship safely. The middle path (capture everything, surface nothing yet) gives us migration immunity without scope bloat.
- Lives in: `02_Systems/QuietKeeper_Memory.md` (Memory state table; MVP scope section). The Codex MVP brief for this system, when written, should call out the "capture-but-don't-surface" pattern explicitly so reviewers don't flag the unused fields as dead code.
- Reversibility: trivial. If a captured field turns out to be wrong-shaped later, the migration cost is not larger than it would have been if we'd never captured it — and most of the time it'll just be ready to use.

2026-04-28 — Title v2 MVP-2: manual equip respects player choice forever

Auto-equip-highest becomes the first-time-only behavior for players who haven't opened the menu yet. 2026-04-28 — Title v2 follow-up: TitleService becomes a ModuleScript, NameTagScript requires it directly
- Decision: convert `TitleService.server.lua` (Script class) into a `TitleService.lua` ModuleScript (with a small `TitleServiceInit.server.lua` Script that calls `Start()`), and have NameTagScript `require` the module directly instead of looking up `_G.SadCaveTitleService`.
- Why: PR #12 used the `_G` shared-state pattern as a quick path during MVP-1. It works, but it's order-sensitive — if NameTagScript's `applyNameTag` runs for an early-joining player before TitleService finishes top-level setup, the nametag silently shows the default payload for that one frame. Luau's `require` guarantees the module's body runs to completion before returning, so the order question vanishes by construction. Cleaner architecture, fewer global-namespace surfaces, and the loud-failure mode (require fails → script errors visibly) is the desired behavior. Doing it now keeps it from compounding into MVP-2's bigger surface (where manual equip + remotes + migration would all also touch the same shared state).
- Alternatives considered: (a) leave the `_G` pattern in place — rejected, it's the kind of fragility that gets harder to back out of as more callers attach to it. (b) keep the runtime fallback to defaults inside NameTagScript so a require failure still renders something — rejected, silent fallback is exactly what made the order-sensitivity hard to spot in the first place; loud failure surfaces bugs faster. (c) add some sort of "wait for TitleService ready" signal — rejected, that's a busy-wait pattern when `require` already gives us the same guarantee for free.
- Lives in: `src/ServerScriptService/TitleService.lua` (ModuleScript) + `TitleServiceInit.server.lua` (Script) + `NameTagScript.server.lua` (require call). PR #13.
- Reversibility: easy — `git revert` the merge commit. Behavior is identical to PR #12 by design, so the rollback is purely architectural.
- Side-finding worth remembering: changing a file's class (Script → ModuleScript) doesn't make Rojo delete the stale live instance in Studio. Codex caught this and manually deleted the orphan `TitleService` Script in Studio. Useful pattern for any future class-swap work — Rojo only adds/updates, doesn't reach across class boundaries.

- Why: prior to today, plan execution state was tracked in three places — the index Plans section, the inline note in the index Active Focus, and (sometimes) the brief filename — none of which were consistent. Putting Status on the plan file itself makes the brief self-describing and gives integration a single field to validate.
- Alternatives considered: (a) keep status only in the index — rejected, drift between index and plan files was already happening. (b) Use Obsidian frontmatter (`status: shipped`) instead of an inline `**Status:**` line — rejected because Obsidian frontmatter doesn't render in plain markdown previews and the plan files want to be readable as plain text.
- Lives in: every file in `06_Codex_Plans/`; `_Plan_Template.md`; `00_Index.md` Plans & Logs section; `AGENTS.md` End-of-Session bookend step 6.
- Reversibility: trivial — strip the line if the convention isn't earning its keep.

2026-04-27 — Decisions register starts here
- Decision: introduce `_Decisions.md` as a forward-looking design-decision register, separate from `_Change_Log.md` (which tracks shipped changes) and `09_Open_Questions/` (which tracks unresolved decisions).
- Why: the change log mixes "we shipped X" with "we picked approach A over B" entries. Searching for *why we chose what we chose* is harder than it needs to be. A separate file with one consistent shape makes the design-history searchable as its own thing without bloating the change log.
- Alternatives considered: (a) keep everything in change log with tighter formatting — rejected because the two genres really do answer different questions; (b) embed decisions inline in system specs only — rejected because cross-cutting decisions (workflow, environments, conventions) don't have a single system spec home; (c) backfill the existing change log into a decisions file — rejected as scope creep, not worth retroactive reclassification.
- Lives in: this file; pointer added to `AGENTS.md` "End of session — Wrap-up" routine, step 3.
- Reversibility: trivial — collapse back into change log if the file isn't earning its keep after a few months.

2026-04-27 — Vault meta-doc structure: keep stale specs as Historical / Superseded
- Decision: when a system gets deleted (e.g. `LevelLeaderstats`, `DailyRewardsServer`), keep its `02_Systems/` spec in place with a Superseded or Removed banner instead of deleting the file.
- Why: the change log, sister specs, and other meta-docs reference these systems by their doc names (e.g. `[[Level_System]]` from `XP_Progression`). Deleting the file would break wikilinks and lose the audit trail. A short "this is historical" banner at the top makes the staleness obvious without breaking the link graph.
- Alternatives considered: delete the file outright — rejected (breaks links); move to a `_archived/` folder — rejected (still breaks `[[Wikilink]]` resolution unless every reference also updates).
- Lives in: `Level_System.md`, `Daily_Rewards.md`, `Level_Stat_System.md` (stub), and reflected in `00_Index.md`'s "Historical / Superseded" section.
- Reversibility: trivial — delete the file later if the audit-trail value fades.
