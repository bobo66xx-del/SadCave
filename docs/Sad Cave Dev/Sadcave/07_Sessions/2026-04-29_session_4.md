## 🤔 Decisions

Five entries added to `_Decisions.md` (one consolidated iteration-1 entry plus four design-call entries from the original design pass):

- **Iteration-1 deltas consolidated** — five amendments to Brief A captured inline in the brief and in the relevant `02_Systems/` specs. Same-branch follow-up commit pattern. Why: Tyler's playtest after Codex's first push surfaced specific tuning issues; same-branch iteration kept the review-blast-radius small and avoided a second PR cycle for what's effectively five tweaks.
- **Sizing client-side** — desktop sizing applies on the client because each viewer's platform determines what they see, not the viewed player's.
- **Stillness fade as thesis-move** — UI behavior tied to "presence rewards stillness" core. Asymmetric thresholds (2.0s out / 0.4s back) deliberately bias the fade-trigger toward "actually running."
- **Split into Brief A + Brief B** — review blast radius and held-item revisit conditions favored two coherent briefs over one mega-brief or eleven micro-briefs.
- **Proportional desktop bump (Option A)** — bumping just the title row would have made the title louder relative to the name, breaking the title-as-epigraph proportion.

## 📝 Notes

- **Four sessions in one day.** Session 1 (AchievementTracker design + brief + Codex review), session 2 (PR #20 backfill + polished-pass design + brief), session 3 (polish-pass review + merge + integration), session 4 (desktop refinement design + two-iteration build + merge + integration). Same loop, four rounds. The cadence is sustainable when each session has a clear scope; this one was about as long as I'd want a single session to run.
- **The two-iteration same-branch loop worked.** Codex's first push matched the brief; live review caught real tuning issues; iteration-1 incorporated them on the same branch; second review confirmed; Tyler merged. The iteration-1 brief amendment pattern (inline value updates + § "Iteration 1 — Review Feedback" at end) means the brief stays the canonical source for Codex without spawning a v2 file. That pattern is worth keeping for any future "small follow-on tunings before merge" cases.
- **Stillness fade is the design highlight.** UI behavior tied to the game's core thesis ("presence rewards stillness") is the kind of move most Roblox games never make. Worth flagging in retrospective if we revisit. The asymmetric 2.0s out / 0.4s back hysteresis is deliberately biased — held titles return easily, fade-out commits only when the player has actually been running.
- **Validation limits worth respecting.** Distance fade, notification dot trigger end-to-end, and tactile press feedback could not be exercised through MCP (Studio's third-person camera overrides CFrame writes; `FireClient` is server-only from execute_luau; mouse-click simulation isn't available). Static review covered all three tightly; Tyler verifies in normal play. Pattern from session 3 carries forward: code reads correct + static + runtime object checks pass ≠ "I clicked it and it felt right," and the latter is Tyler's call.
- **Pushed back against Tyler's "all 11" framing.** When he greenlit every refinement option I'd surfaced in the design critique, I split into Brief A + B and held two items entirely. Honest pushback was the right move — saying yes to all 11 in one brief would have inflated the review surface and bundled "wait and see" items with high-impact ones. Same instinct as session 3's "extend cleanup scope when actively breaking new behavior" decision: scope discipline matters even when the user is enthusiastic.
