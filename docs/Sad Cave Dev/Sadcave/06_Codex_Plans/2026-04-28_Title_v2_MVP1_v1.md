## 1. Purpose

Build the **data + display layer** of Title v2 — the cosmetic identity layer for Sad Cave. After this brief, players see their highest-earned level title under their name on the nametag, and milestone level-ups display a combined `level N — new title: X` fade above the XPBar. No menu, no manual equip, no migration of v1 data, no other title categories firing yet.

This is the first of two MVP slices. **MVP-2** (separate later brief) adds the placeholder TitleMenu, manual equip/unequip handlers + remotes, and production-cutover migration code. The **polished TitleMenu** is its own brief after that, designed by Tyler in a separate focused session.

The slate is genuinely empty for this work: the v1 title pipeline (`TitleService`, `TitleConfig`, `TitleMenu`, `TitleEffectPreview`, plus the title-rendering side of the old `NameTagScript Owner`) was deleted in the 2026-04-27 cleanup. Cowork session 7 Reality Check (2026-04-28) confirmed no Title* instances remain in `ReplicatedStorage` or `ServerScriptService`. There is one forward-looking TODO already in place: line 220 of `XPBarController.client.lua` carries `-- TODO: Listen to TitleRemotes.TitleDataUpdated for new-title fades when TitleService v2 ships.` — that's the integration point, not drift.

## 2. Player Experience

A player joins the testing place. Under their name on the nametag, lowercase, in a soft warm color, sits their auto-equipped title — `still here` if they're around level 10, `cathedral dark` at 500. The title row reads smaller and softer than the name above it. For titles with a `tint` effect, the text is statically tinted with the title's `tintColor`. For `shimmer`/`pulse`/`glow` titles, a subtle per-frame animation runs on the client.

As they play and cross a milestone level, a single fade appears above the XPBar reading `level N — new title: X` (e.g. `level 50 — new title: familiar face`). Holds 5 seconds, fades out. The nametag title updates to the new auto-equipped title. Milestone levels are: 1, 5, 10, 20, 35, 50, 75, 100, 150, 200, 300, 400, 500, 700, 1000, 1500, 2000, 3000, 5000, 10000 (per the level milestones table in `Title_System.md`).

A level-up that doesn't cross a title milestone (e.g. level 23 → 24) shows only `level N` for 5 seconds — same animation as currently shipped, just longer hold. This bumps the existing animation from its current 2s display window to 5s.

Players who own gamepass `1797105034` (the cosmetic titles gamepass — distinct from the XP-multiplier gamepass `1790063497`) have all six gamepass titles in their owned set. **Auto-equip tier ordering:** the spec doesn't define gamepass-vs-level priority. For MVP-1, **rank gamepass titles above level titles** when picking the auto-equip; this gives gamepass owners visible value without requiring manual choice yet. (When MVP-2 lands manual equip, players choose freely and this priority becomes irrelevant.)

The XPBar's hover-to-show-progress behavior is **unchanged** — the 2-second hover reveal stays. Only the unlock-notification animations (`playLevelUp`) move to 5s.

**What does NOT happen yet** (deferred to MVP-2 or later): no title menu, no manual title switching, no migration of `EquippedTitleV1`, no achievement/presence/exploration/seasonal categories firing, no polished UI.

## 3. Technical Structure

### Server responsibilities

- **`TitleConfig`** — new ModuleScript at `ReplicatedStorage.TitleConfig`. Exports `{ titles = { [id] = {...} }, MIGRATION = {...} }` with all 60 title definitions baked in. Active categories on first ship: `level` + `gamepass`. Other categories (`exploration`, `presence`, `achievement`, `seasonal`) defined but not actively checked yet — keeps shape stable for later briefs.
- **`TitleService`** — new ServerScript at `ServerScriptService.TitleService`. Responsibilities:
  - Initial ownership resolution + auto-equip on `Players.PlayerAdded`. Reads `leaderstats.Level.Value` (created by ProgressionService) and gamepass ownership via `MarketplaceService:UserOwnsGamePassAsync(player, 1797105034)` (cached per-player).
  - Listens to `Player.leaderstats.Level:GetPropertyChangedSignal("Value")` to detect level changes. (We can't modify `ProgressionService` — it's no-touch and is what fires the `LevelUp` RemoteEvent on line 409. Observing leaderstats is the non-invasive path.)
  - On level change: re-resolve ownership; if a higher-tier title is now owned, auto-equip it; fire `TitleRemotes.TitleDataUpdated` with the new equipped state and a `newlyUnlocked = true` flag.
  - On gamepass purchase via `MarketplaceService.PromptGamePassPurchaseFinished`: refresh ownership cache, re-resolve, possibly auto-equip.
  - DataStore: writes to a new `TitleData` per-player key with shape `{ equippedTitle = "id_string", achievements = {} }`. The `achievements` field is declared but unused in MVP-1 — keeps shape stable for the AchievementTracker brief later. Uses pcall + retry on transient DataStore errors, same pattern as ProgressionService. Falls back to in-memory state on persistent failure.
  - On player leave: persist current `equippedTitle` to `TitleData`.
- **`NameTagScript`** modifications — bump BillboardGui height; add a TitleLabel; set attributes on the BillboardGui that the client effect controller reads.

### Client responsibilities

- **`NameTagEffectController`** — new LocalScript at `StarterPlayer.StarterPlayerScripts.NameTagEffectController.client.lua`. Watches the workspace for `BillboardGui` instances named `NameTag` appearing on character `HumanoidRootPart`s. For each, reads `TitleEffect` attribute (`none` | `tint` | `shimmer` | `pulse` | `glow`) and `TitleTintColor` attribute (Color3) on the BillboardGui, finds the `TitleLabel` child, and runs the appropriate animation:
  - `none` — set `TextColor3` = warm-white default; no animation.
  - `tint` — set `TextColor3` = TitleTintColor; no animation.
  - `shimmer` — slow left-to-right color sweep using TitleTintColor at low opacity. Period ~3s. Use `RunService.Heartbeat` or a `TweenService` loop.
  - `pulse` — gentle brightness oscillation of TitleTintColor, ~3s cycle. `TweenService` ping-pong.
  - `glow` — soft constant glow effect — TitleLabel TextColor3 = TitleTintColor + a small `UIStroke` or low-opacity duplicate label behind it for the glow halo. Static.
  - When `TitleEffect` attribute changes (player equips a different title), restart the animation under the new effect.
- **`XPBarController`** modifications — extend `playLevelUp` to render the combined fade format and 5s hold; listen to `TitleRemotes.TitleDataUpdated`; client-side coalesce LevelUp + TitleDataUpdated into combined fade.

### Remote events / functions

- **New `ReplicatedStorage.TitleRemotes/`** Folder.
- **New `ReplicatedStorage.TitleRemotes.TitleDataUpdated`** RemoteEvent. Payload: `{ equippedTitle = "id", equippedDisplay = "still here", equippedEffect = "tint", equippedTintColor = Color3, newlyUnlocked = boolean }`. Server fires; client listens.
- **Existing `ReplicatedStorage.Progression.LevelUp`** RemoteEvent — **unchanged**. ProgressionService is no-touch. Coalescing happens on the client in `XPBarController`.

### DataStore keys touched

- **New `TitleData`** key — written by `TitleService` per-player. Read on join. Pcall + retry pattern matching ProgressionService.
- **Not touched:** `ProgressionData` (no-touch, owned by ProgressionService), `EquippedTitleV1` (deferred to MVP-2 migration), `NoteSystem`/`CurrentNotes` (unrelated), `Game__OFFICIAL__Reports`/`Reports97` (unrelated, no-touch).

### Client-side LevelUp + TitleDataUpdated coalescing

This is the trickiest piece architecturally. Both events fire from the server in the same tick when crossing a title milestone. On the client, they arrive in close sequence (LevelUp first because ProgressionService.lua line 409 fires it before TitleService observes the leaderstats change and fires TitleDataUpdated). The XPBar coalesces them with a small defer:

```
on LevelUp(payload):
    capture newLevel from payload
    schedule a deferred render in 100ms:
        if a TitleDataUpdated with newlyUnlocked=true arrived for this player in the last 100ms:
            render combined fade "level N — new title: X" with 5s hold
        else:
            render level-up only "level N" with 5s hold

on TitleDataUpdated(payload):
    if payload.newlyUnlocked and a recent LevelUp is pending:
        merge into the pending level-up render (combined fade)
    else if payload.newlyUnlocked:
        render new-title-only fade "new title: X" with 5s hold
        (this path is rare in MVP-1 — fires on gamepass purchase)
    else:
        update nametag display attributes only, no fade (this path is for equip changes, used in MVP-2)
```

The 100ms window is comfortably larger than the actual server-side firing gap (which is microseconds to a few ms in practice) and small enough not to feel laggy. Use `task.delay(0.1, ...)` with a token pattern matching the existing `revealToken` in `XPBarController.client.lua`.

## 7. Security / DataStore Notes

- ⚠️ **Server-authoritative ownership.** TitleService computes ownership server-side from authoritative inputs (leaderstats.Level, MarketplaceService gamepass check). Clients never claim ownership. In MVP-1 there is no client-driven equip path, so this is straightforward; MVP-2 will add EquipTitle validation but that's a later concern.
- ⚠️ **DataStore retry/pcall.** Use the same pattern as ProgressionService — pcall, retry up to 3 times with `task.wait(1)` backoff, warn on persistent failure, fall back to in-memory state. Don't block the join flow on DataStore — start with default `equippedTitle = "new_here"` if load fails, then attempt save on next equip change.
- ⚠️ **Gamepass cache.** `MarketplaceService:UserOwnsGamePassAsync` is a network call. Cache per-player on join and on `PromptGamePassPurchaseFinished`. Don't poll.
- ⚠️ **Rate limits.** TitleData writes only on equip change (rare — at most once per milestone level-up + once per gamepass purchase + once on player leave). No per-tick writes. Consume `BUDGET.SetIncrementAsync` request budget conservatively.
- ⚠️ **No client trust.** TitleDataUpdated payload is server-generated. Client renders what server says; never the other way around.

## 10. Rollback Notes

- All five new files (`TitleConfig.lua`, `TitleRemotes/init.meta.json`, `TitleRemotes/TitleDataUpdated.meta.json`, `TitleService.server.lua`, `NameTagEffectController.client.lua`) are additive. Removing them disables the feature entirely.

- `XPBarController.client.lua` modifications are timing constants + new event listener + new combined-fade format. Revert with `git revert`.
- DataStore writes to `TitleData` are forward-only — no destructive overwrites of other keys. If the feature is rolled back, `TitleData` records persist but become unread (no harm).
- If a player ends up with a corrupted `TitleData.equippedTitle` (an ID that no longer exists in TitleConfig), TitleService should fall back gracefully to auto-equip-highest. Document this fallback in the service code.
