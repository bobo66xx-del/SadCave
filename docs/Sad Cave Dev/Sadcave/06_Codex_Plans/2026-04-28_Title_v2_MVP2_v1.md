## 1. Purpose

Layer the **manual-control + menu surface + production-cutover migration** on top of the data + display layer that PR #12 + PR #13 shipped. After MVP-2, players can browse their owned and locked titles in a (deliberately minimal) menu, manually equip any owned title, and have that choice survive future level-ups. The placeholder menu is functional but visually minimal — the polished v2 menu is a separate later brief that drops in over the placeholder. Production cutover migration code ships in this brief so the production place can flip to v2 with one config change later; the testing place won't actually exercise the migration code (no v1 data exists there) but it's worth shipping coherently with the manual-equip surface that creates the data shape it migrates into.

This is the **second of two MVP slices** for Title v2. After this ships, Title v2's player-facing surface is feature-complete except for the polished menu pass; later briefs add achievement/presence/exploration/seasonal categories on top of the same data layer without disturbing this work.

## 2. Player Experience

A player who already had MVP-1 sees their auto-equipped highest title on the nametag (e.g. `believer` for gamepass owners, `cathedral_dark` for level 500+ without gamepass). They look at a screen corner and see a small lowercase `titles` text button. Clicking it opens a small dark menu with two tabs at the top: `owned` and `locked`.

- **Owned tab:** scrollable list of every title they have. Each row shows the title's display text in lowercase, a small color dot on the left tinted to the title's `tintColor`, and (on the currently equipped row) a subtle dot or marker indicating "this is what's worn." Clicking any owned row equips that title — the nametag updates within a second, the menu re-renders to show the new equipped marker, and the change persists across rejoins.
- **Locked tab:** scrollable list of every title they don't yet own. Each row shows the display text + color dot + a short hint of how to unlock it ("reach level 50", "play for 24 hours", "complete your first conversation", etc.). No interaction — locked rows aren't clickable.

A player who manually equipped, say, `slow_steps` at level 35 then later levels up to 50 (which would auto-equip `familiar_face` for a default player) keeps `slow_steps` equipped. The combined level-up + new-title fade still fires above the XPBar (`level 50 — new title: familiar face`) so they know about the unlock; they can open the menu and switch if they want. Their manual choice is respected forever until they explicitly change it again or hit an "unequip" path that lets auto-equip-highest take over (clicking the currently-equipped row toggles back to auto-equip, surfacing the highest owned title).

A returning **production** player whose old v1 title was equipped (e.g. `regular`, `socialite`, `icon`, `legend` from the v1 list) sees their title migrated to the nearest v2 equivalent on first join under v2 (e.g. `regular` → `familiar_face`, `socialite` → `half_awake`, `icon` → `cathedral_dark`, `legend` → `nowhere_else`). One read, one write to the new `TitleData` key, then `EquippedTitleV1` is never touched for that player again. If their v1 title was a shop-purchased one with no v2 equivalent (e.g. shards-bought titles), they reset to `new_here` and pick up auto-equip-highest from there.

**What does NOT happen yet** (deferred to later briefs): the polished TitleMenu visual design (this is the placeholder), the achievement / presence / exploration / seasonal title categories firing (TitleConfig has the data, but ownership resolution still only checks level + gamepass — those briefs come later), and the v1 retirement (the migration code stays in place until v2 has been stable in production for some time).

## 3. Technical Structure

### Server responsibilities — TitleService extension

All changes land in `src/ServerScriptService/TitleService.lua` (the ModuleScript from PR #13). Public API gains two new functions; internal state gains one new field; the auto-equip logic gains one new branch.

- **New field on `playerStates[userId]`:** `equippedManually` (boolean, defaults false). Persisted to `TitleData.equippedManually`.
- **New field on `playerStates[userId]`:** `migratedFromV1` (boolean, defaults false). Persisted to `TitleData.migratedFromV1`. Once true, never re-read `EquippedTitleV1` for that player.
- **New private function `equipTitle(player, titleId)`:** validates the player owns the title (re-resolves ownership server-side, checks `ownedSet[titleId]`), then sets `state.equippedTitle = titleId`, sets `state.equippedManually = true`, fires `TitleDataUpdated` to the client with `newlyUnlocked = false` (this is a manual equip, not a fresh unlock — the unlock fade should NOT fire), updates the nametag, persists. Returns true on success, false + warn on failure (title not owned, title id unknown, etc.).
- **New private function `unequipTitle(player)`:** clears `state.equippedManually = false`, then calls `refreshAutoEquip(player, false)` to let auto-equip-highest take over. Persists. Fires `TitleDataUpdated` via the refresh path.
- **Modified `refreshAutoEquip(player, newlyUnlocked)`:** add an early branch — if `state.equippedManually` is true, do NOT change `state.equippedTitle`. Still fire `TitleDataUpdated` with `newlyUnlocked = true` if the *available set* changed (so the player gets the unlock notification fade), but the equipped title and the nametag stay on the manual choice. Implementation: split the current `refreshAutoEquip` into "compute new ownership + detect new unlock" and "apply auto-equip if not manually controlled" steps.
- **New RemoteEvent listeners:** `EquipTitle.OnServerEvent:Connect(function(player, titleId) equipTitle(player, titleId) end)` with rate-limit guards (one equip per N seconds per player; suggest 1 second min) and basic validation (titleId is a string, exists in TitleConfig). `UnequipTitle.OnServerEvent:Connect(function(player) unequipTitle(player) end)` with same rate-limit guard.
- **Updated `loadTitleData(player)`:** after loading the `TitleData` key, if `result.migratedFromV1` is not true AND `result` is missing `equippedTitle` (or this is a fresh `TitleData` with no entry), attempt the v1 migration flow (see "Migration" below).
- **Updated `saveTitleData(player)`:** writes the new fields `equippedManually` and `migratedFromV1` alongside `equippedTitle` and `achievements`.

### Server responsibilities — Migration

A new private function `migrateFromV1(player, currentTitleData)` that runs once per player, on first join under v2:

1. Open the `EquippedTitleV1` DataStore via `DataStoreService:GetDataStore("EquippedTitleV1")`. (If the v1 codebase used a different DataStore name, Codex should grep the historical repo state at the pre-2026-04-27-cleanup tag for the actual name. If it can't be determined cleanly, pick the documented name from the spec's migration plan — `EquippedTitleV1` — and log a warn if the read fails.)
2. `:GetAsync(tostring(player.UserId))` with pcall + retry (3 attempts, 1s backoff, same pattern as the rest of TitleService).
3. If the read returns a string that is a key in `TitleConfig.MIGRATION` → set `currentTitleData.equippedTitle = TitleConfig.MIGRATION[v1Title]`. If the read returns a string that's NOT in the migration table (shop title, retired ID, etc.) → leave `equippedTitle` as `TitleConfig.DEFAULT_TITLE_ID`. If the read returns nil or fails → leave `equippedTitle` as `TitleConfig.DEFAULT_TITLE_ID` (fresh player on this account).
4. Set `currentTitleData.migratedFromV1 = true` regardless of whether a v1 title was found, so the read happens at most once per player. Set `equippedManually = false` (the migrated title is treated as auto-equipped, not as a deliberate manual choice — players who want it locked in can re-equip via the menu).
5. Persist via the normal `saveTitleData` path during the join flow.

The migration is **additive and safe** — it never overwrites an existing `TitleData.equippedTitle`. It only fires when `migratedFromV1` is false AND the player has no prior v2 data. The testing place's existing test player (Tyler at level 558+ with `believer` equipped via gamepass) already has `migratedFromV1 = false` AND existing `TitleData`, so the migration code will not fire for him — he keeps his current state. Production players who are joining for the first time post-cutover are the only ones who actually exercise this code.

### Client responsibilities — Placeholder TitleMenu

A new ScreenGui at `src/StarterGui/TitleMenu/` with one LocalScript controlling it. Visually minimal — intentionally not polished. The goal is functional, ugly-but-not-embarrassing.

**Structure:**
- `ScreenGui` named `TitleMenu` (`ResetOnSpawn = false` so the menu state survives respawns; `IgnoreGuiInset = true`).
- A root `Frame` named `Root` — dark background (e.g. `BackgroundColor3 = Color3.fromRGB(20, 18, 22)`, `BackgroundTransparency = 0.1`), centered, sized for ~70% of the viewport (e.g. `Size = UDim2.new(0.7, 0, 0.7, 0)`, `AnchorPoint = Vector2.new(0.5, 0.5)`, `Position = UDim2.new(0.5, 0, 0.5, 0)`). `Visible = false` by default — the toggle button (see below) flips this.
- A header `Frame` containing two `TextButton` tabs labeled `owned` and `locked` (lowercase, Gotham 14, soft warm color), and a small `TextButton` labeled `×` in the top-right that flips `Root.Visible = false`.
- Below the header, a `Frame` named `Body` containing two child `ScrollingFrame`s: `OwnedScroll` and `LockedScroll`. Tab clicks toggle `.Visible` between the two; only one is visible at a time.
- Each scroll uses a `UIListLayout` (Vertical, padding ~4px) and a `UIPadding` for inset.
- Each title row is a `Frame` (or a `TextButton` for owned, plain `Frame` for locked):
  - `ColorDot` — a small `Frame` with `BackgroundColor3 = title.tintColor`, sized ~12×12 with rounded corners (or a UICorner).
  - `TitleText` — a `TextLabel` with the title's `display` in lowercase, Gotham 12, soft warm color, left-aligned.
  - For locked rows: a `HintText` `TextLabel` on the right, gray-italic style, with a short unlock hint (e.g. "reach level 50", "buy the gamepass", "play for 24 hours", "complete your first conversation"). Hint generation rules below.
  - For owned rows: an `EquippedDot` `Frame` on the right that's only visible when this row's `titleId == state.equippedTitle`. Subtle indicator — small filled circle in the title's tintColor.

**Hint generation rules** (client-side, deterministic from TitleConfig):
- `category == "level"` → `"reach level " .. title.levelRequired`
- `category == "gamepass"` → `"buy the title pack"`
- `category == "presence"` → `"play for " .. title.hoursRequired .. " hours"` (singular "1 hour" if hoursRequired == 1)
- `category == "exploration"` → `"discover " .. title.zoneId:gsub("_", " ")`
- `category == "achievement"` → for now, fallback to `"do something special"` since AchievementTracker isn't built yet — OK because none of these will appear in the locked tab unless a future brief activates the category. (Codex can wire a small hint-lookup table for the achievement IDs if it's cleaner than the fallback.)
- `category == "seasonal"` → `"during a special event"`

**Click handler (owned rows):**
1. If clicked title is the currently equipped title → fire `UnequipTitle:FireServer()` (toggles back to auto-equip-highest). The next `TitleDataUpdated` from the server will rerender the menu.
2. If clicked title is a different owned title → fire `EquipTitle:FireServer(titleId)`. The next `TitleDataUpdated` rerenders.

**Listener:**
- Subscribe to `TitleRemotes.TitleDataUpdated` — when the payload's `equippedTitle` changes, update the equipped indicator in the rendered list (don't re-create the whole list, just toggle the marker visibility).

**Open / close:**
- Listen for the `TitlesToggleButton` ScreenGui's button click (see next section) — toggles `Root.Visible`.
- Optional: also bind ESC key to close (only when the menu is open). Skip if it adds friction.

**Initial render:**
- On player join (after `TitleDataUpdated` first fires), build the full list of all 56 titles from `TitleConfig.titles`. Determine each title's owned/locked state by checking against an additional payload field — see "Wire surface adjustments" below.

### Client responsibilities — TitlesToggleButton

A separate small ScreenGui at `src/StarterGui/TitlesToggleButton/`. Single TextButton, lowercase `titles` text, soft warm color, dark transparent background, anchored to a screen corner.

**Suggested placement:** top-right corner, ~60px from the right edge and ~24px from the top, sized ~80×24px. Reasoning: top-left and bottom-left are typically used by Roblox's chat / leaderstats / mobile thumbstick; bottom-right is Roblox's PlayerList / settings; top-right is the most consistently free corner across desktop and mobile. If `Game Version` ScreenGui (Tyler kept this Studio-only) is in the top-right, place `titles` just to its left or below it. Codex should `inspect_instance` `Game Version` placement before finalizing.

**Tone constraints:**
- Lowercase only — no emoji, no all-caps, no decoration.
- Soft warm color matching the rest of Sad Cave's UI palette — `Color3.fromRGB(225, 215, 200)` is the established default.
- Background semi-transparent dark — should read as ambient, not as an obvious clickable button. Hover/press states allowed but subtle (small color shift, no scaling animations).
- Should not overlap or interact with the XPBar visually.

**Click handler:** fires a `BindableEvent` (or sets a shared attribute) that the TitleMenu's LocalScript reads to toggle `Root.Visible`. Codex's choice — a simple BindableEvent in `ReplicatedStorage` or a `Players.LocalPlayer:GetAttribute` flag works either way; the BindableEvent pattern is cleaner if the polished menu later wants to add open/close hooks.

### Wire surface adjustments

`TitleRemotes.TitleDataUpdated` payload **gains one optional field**:

```lua
{
    equippedTitle = "id",
    equippedDisplay = "still here",
    equippedEffect = "tint",
    equippedTintColor = Color3,
    newlyUnlocked = boolean,
    -- NEW in MVP-2:
    ownedTitleIds = { "new_here", "settling_in", "still_here", ... },  -- list of all title ids the player currently owns; client uses this to render owned/locked tabs
}
```

The `ownedTitleIds` field is fired on every `TitleDataUpdated` event (join, level milestone, gamepass purchase, equip, unequip). Adds a tiny amount of bandwidth (50 strings max) but keeps the menu's owned/locked state in perfect sync with the server without requiring a separate "ownership query" remote. If bandwidth is a concern in production scale, swap to a "diff" pattern (only send adds/removes); for MVP-2 the full list is simpler.

`TitleRemotes/` gains two new RemoteEvents:
- `TitleRemotes.EquipTitle` — RemoteEvent. Client → server. Payload: `titleId` (string).
- `TitleRemotes.UnequipTitle` — RemoteEvent. Client → server. No payload.

### DataStore — TitleData shape

`TitleData` shape evolves from MVP-1's `{ equippedTitle, achievements }` to:

```lua
{
    equippedTitle = "still_here",        -- existing, unchanged
    achievements = { ... },               -- existing, unchanged (still empty in MVP-2; AchievementTracker brief later)
    equippedManually = false,             -- NEW: true if the player explicitly equipped via menu, false if auto-equipped
    migratedFromV1 = false,               -- NEW: true once v1 migration has been attempted (success or no-op), false on a fresh slate
}
```

Backward-compatible with MVP-1 reads — the old shape will load with the two new fields defaulting to `false`. Forward-compatible with future fields (achievements get filled in once AchievementTracker ships).

Same key (`TitleData`), same DataStore (`titleStore`), same pcall + retry pattern.

## 7. Security / DataStore Notes

- ⚠️ **Server-authoritative ownership for equip.** `EquipTitle.OnServerEvent` MUST re-resolve ownership server-side before applying. Never trust the client's claim that they own a title — always verify against the current ownedSet computed from leaderstats.Level + gamepass cache. If the client requests a title they don't own, warn-and-no-op (don't crash, don't echo back, just log a warn — this could be a benign race or an exploit attempt).
- ⚠️ **Rate-limit equip/unequip.** Naive client could spam EquipTitle. One equip per second per player is plenty — block subsequent calls within the window. Same for unequip.
- ⚠️ **Pcall + retry on TitleData writes** — same pattern as MVP-1.
- ⚠️ **Migration is one-shot per player.** Do NOT re-read `EquippedTitleV1` if `migratedFromV1` is already true. Production cutover plan eventually drops the migration code entirely; for now it's defensive read-once-and-flag.
- ⚠️ **`ResetOnSpawn = false` on TitleMenu** — without this, the menu disappears on every respawn, which is jarring. Same for `TitlesToggleButton`.
- ⚠️ **No client-driven ownership claims.** The client receives `ownedTitleIds` from the server and renders accordingly. The server never trusts a client-side "I own this" assertion.
